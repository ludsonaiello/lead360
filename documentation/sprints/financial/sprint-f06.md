## Purpose

Service businesses have predictable recurring costs that repeat every month, every week, or every year — insurance premiums, vehicle registrations, software subscriptions, equipment leases, storage units, recurring tool rentals. These expenses exist whether or not any project is active. Right now they must be manually entered every cycle. Missed entries mean incomplete financials.

The Recurring Expense Engine automates this. A business owner defines a rule once: "Monthly insurance premium, $1,850, first of every month, Chase Business Checking, auto-confirm." The BullMQ scheduler runs nightly, checks which rules are due, and generates confirmed expense entries automatically. Rules that require human review generate `pending_review` entries instead — the bookkeeper approves them in the morning.

The generated entries are first-class `financial_entry` records — they appear in all reports, exports, and P&L calculations exactly like manually created entries. The only difference is the `is_recurring_instance = true` flag and the `recurring_rule_id` link back to the rule that generated them.

## Scope

### In Scope

- New `recurring_expense_rule` table with full rule definition
- Full CRUD endpoints for recurring rules
- Preview endpoint: show upcoming obligations for next 30/60/90 days without generating entries
- BullMQ scheduled job: nightly run, checks all active rules, generates due entries
- Manual trigger endpoint: force-generate an entry for a specific rule immediately (for testing or catch-up)
- Skip endpoint: skip the next occurrence of a rule without disabling the rule
- Rule pause/resume: temporarily disable a rule without deleting it
- History endpoint: list all entries generated by a specific rule
- `is_recurring_instance` and `recurring_rule_id` fields on `financial_entry` already exist from F-01 — no new schema migration needed on that table
- 100% API documentation
- Full test coverage including scheduler behavior

### Out of Scope

- No frontend implementation
- No notification to users when recurring entries are generated (deferred to a future notification sprint)
- No approval workflow integration beyond what F-04 already provides (`submission_status` on generated entries)
- No support for variable amounts (rules have a fixed amount — variable billing deferred)
- No external billing integration (Stripe, QuickBooks recurring) — rules are internal only
- No changes to any table other than adding `recurring_expense_rule`

## Next Due Date Calculation Logic

This is the core scheduling algorithm. It must be implemented as a pure, unit-testable function in `RecurringExpenseService.calculateNextDueDate()`.

**Inputs:** `frequency`, `interval`, `day_of_month`, `day_of_week`, `current_due_date`

**Output:** The next `DateTime` after `current_due_date` when the rule is due.

**Rules by frequency:**

`daily`: Add `interval` days to `current_due_date`.

`weekly`: Add `interval * 7` days. If `day_of_week` is set, find the next occurrence of that weekday at or after `current_due_date + interval*7`.

`monthly`: Add `interval` months. If `day_of_month` is set, use that day of the calculated month. If the calculated month does not have that day (e.g., `day_of_month=31` in February), use the last day of that month. Never overflow to the next month.

`quarterly`: Add `interval * 3` months. Same `day_of_month` logic as monthly.

`annual`: Add `interval` years. Same `day_of_month` logic.

**Edge case — end of month:** A rule created on January 31 with `monthly` frequency and no `day_of_month` uses `start_date` day (31). February has no day 31 — use February 28 (or 29 in leap years). March 31 is valid — use March 31. The rule snaps to end-of-month when the target day does not exist.

## BullMQ Scheduler Architecture

### Scheduler

Location: `api/src/modules/jobs/schedulers/recurring-expense.scheduler.ts`

Pattern: Follow existing schedulers in `api/src/modules/jobs/schedulers/`. Read those files before implementing.

**Schedule:** Runs nightly at **02:00 AM server time** using a cron expression: `0 2 * * *`.

**Behavior:**
1. Query all `recurring_expense_rule` records where:
   - `status = 'active'`
   - `next_due_date <= TODAY` (today's date in server local time)
2. For each rule found, enqueue a `recurring-expense-generate` job with the rule ID and tenant ID.
3. The scheduler does not generate entries directly — it only enqueues jobs.

### Queue Name
`recurring-expense-generation`

### Job Payload
```typescript
interface RecurringExpenseJobPayload {
  ruleId: string;
  tenantId: string;
}
```

### Processor

Location: `api/src/modules/jobs/processors/recurring-expense.processor.ts`

**Processor behavior for each job:**

1. Fetch the rule — verify `status = 'active'` and `next_due_date <= TODAY`. If not (rule was paused between enqueue and processing), skip silently and mark job complete.
2. Call `FinancialEntryService.createEntry()` with:
   - `tenantId` from rule
   - `userId` = `created_by_user_id` from rule (entry attributed to rule creator)
   - `userRole` = `'Owner'` (system-generated entries bypass submission_status defaulting — always use `auto_confirm` value from rule)
   - `dto` assembled from rule fields: `category_id`, `amount`, `tax_amount`, `supplier_id`, `vendor_name`, `payment_method_registry_id`, `entry_date = next_due_date`, `notes`, `is_recurring_instance = true`, `recurring_rule_id = rule.id`
   - `submission_status` = `'confirmed'` if `auto_confirm = true`, `'pending_review'` if `auto_confirm = false`
3. After successful entry creation, update the rule:
   - `occurrences_generated += 1`
   - `last_generated_at = NOW()`
   - `last_generated_entry_id = new_entry.id`
   - `next_due_date = calculateNextDueDate(rule)`
4. Check termination conditions — if either is met, set `status = 'completed'`:
   - `end_date` is set and `next_due_date > end_date`
   - `recurrence_count` is set and `occurrences_generated >= recurrence_count`
5. All steps 2–4 must be wrapped in a Prisma transaction — if entry creation fails, the rule update must not persist.

**Retry configuration:** Max 3 attempts, exponential backoff starting at 10 seconds. On final failure, log error with rule ID and mark job complete (do not poison the queue).

#### `GET /financial/recurring-rules`

**Query parameters:**
- `status` — `recurring_rule_status` enum, optional. Default: `active`.
- `category_id` — UUID, optional.
- `frequency` — `recurring_frequency` enum, optional.
- `page` — integer, default 1.
- `limit` — integer, default 20, max 100.
- `sort_by` — `next_due_date` | `amount` | `name` | `created_at`. Default: `next_due_date`.
- `sort_order` — `asc` | `desc`. Default: `asc`.

**Response:**
```
data: [
  {
    id
    name
    description
    category: { id, name, type, classification }
    amount
    tax_amount
    supplier: { id, name } | null
    vendor_name
    payment_method: { id, nickname, type } | null
    frequency
    interval
    next_due_date
    auto_confirm
    status
    occurrences_generated
    recurrence_count       — null if indefinite
    end_date               — null if indefinite
    last_generated_at
    created_at

meta: { total, page, limit, total_pages }
summary: {
  total_active_rules:    integer
  monthly_obligation:    decimal   — estimated monthly cost across all active rules (normalized to monthly)
}
```

The `monthly_obligation` in the summary normalizes all rule frequencies to a monthly equivalent:
- `daily`: `amount * interval * 30`
- `weekly`: `amount * (30 / (interval * 7))`
- `monthly`: `amount / interval`
- `quarterly`: `amount / (interval * 3)`
- `annual`: `amount / (interval * 12)`

This gives the business owner a single number representing their fixed monthly overhead commitment.

All fields optional. Partial update.

**Not editable via PATCH:** `status` (use pause/resume/cancel), `occurrences_generated`, `last_generated_at`, `last_generated_entry_id`, `next_due_date` (recalculated automatically), `is_recurring_instance`.

#### `DELETE /financial/recurring-rules/:id`

Sets `status = 'cancelled'`. Does not hard-delete the record — historical context must be preserved.

Generated entries are NOT deleted — they are independent `financial_entry` records. The `recurring_rule_id` link remains for historical traceability.

**Response:** 200 OK — updated rule with `status = 'cancelled'`.

---

#### `POST /financial/recurring-rules/:id/pause`

Sets `status = 'paused'`. The scheduler skips paused rules. `next_due_date` is preserved — when resumed, the rule picks up where it left off.

**Errors:**
- 400 — rule is not `active`

**Response:** 200 OK — updated rule.

---

#### `POST /financial/recurring-rules/:id/resume`

Sets `status = 'active'`.

**Behavior on resume:**
- If `next_due_date` is in the past (rule was paused and missed occurrences), do NOT back-generate missed entries. Simply update `next_due_date` to the next upcoming occurrence from today.
- This is the correct behavior for service businesses — they do not want automatic retroactive entries for a months-long pause.

**Errors:**
- 400 — rule is not `paused`

**Response:** 200 OK — updated rule with recalculated `next_due_date`.

---

#### `POST /financial/recurring-rules/:id/trigger`

Manually generate the next entry immediately, outside the normal schedule.

**Roles:** Owner, Admin only.

**Behavior:**
1. Verify rule is `active` or `paused`.
2. Enqueue an immediate `recurring-expense-generate` job (priority: high).
3. The job runs the same processor logic as the scheduled run.

**Response:** 202 Accepted — `{ message: "Entry generation triggered", rule_id: "..." }`. The entry is created asynchronously.

**Errors:**
- 400 — rule is `cancelled` or `completed`

---

#### `POST /financial/recurring-rules/:id/skip`

Skip the next scheduled occurrence. The rule advances to the following occurrence without generating an entry for the current `next_due_date`.

**Behavior:**
1. Verify rule is `active`.
2. Calculate the occurrence after `next_due_date` using `calculateNextDueDate()`.
3. Set `next_due_date` to that next occurrence.
4. Increment `occurrences_generated` (the skip counts as an occurrence for `recurrence_count` purposes — this prevents the rule from running forever by skipping every time).
5. Check termination conditions after incrementing.
6. Audit log the skip with reason.

**Request body:**
```
reason   string   optional   max 500 chars — documented in audit log
```

**Response:** 200 OK — updated rule with new `next_due_date`.

## Business Rules Summary

1. Rules cannot have a `start_date` in the past — they start today or in the future.
2. `day_of_month` is capped at 28 to avoid month-end overflow complexity. Users who want end-of-month behavior set `day_of_month = 28`.
3. When a rule is resumed after a pause, missed occurrences are NOT back-generated. `next_due_date` advances to the next upcoming date.
4. A `cancelled` or `completed` rule cannot be edited or reactivated.
5. A `paused` rule can be edited — changes take effect when resumed.
6. Skipping an occurrence counts toward `recurrence_count`.
7. Termination (status → `completed`) is checked after every generation and after every skip.
8. Generated entries are fully independent `financial_entry` records — cancelling a rule does not affect already-generated entries.
9. The `trigger` endpoint generates one entry (the next occurrence) — it does not generate all past-due entries.
10. `auto_confirm = true` → generated entries have `submission_status = confirmed`. `auto_confirm = false` → `pending_review`. The F-04 role-based override does not apply to system-generated entries — the rule's `auto_confirm` setting is authoritative.
11. The scheduler only runs each rule once per `next_due_date` — re-running the scheduler on the same day will not generate duplicate entries (the processor checks `next_due_date <= TODAY` but also checks `last_generated_at` was not already today for that rule).
12. Maximum 100 active recurring rules per tenant. Enforce at service level on create.
